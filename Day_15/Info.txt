// ***************************** __ MAIN POINTS __  ***********************************
0) To use transactions there are couple of counditions to be hold:
    - Mongo run on replica/sharded - transaction mechanism based on opLog file that are not available on single instance.
    - the driver should be WiredTiger and not some customization of it.
    - each transaction should include session (start/close, transferring to the DB)
1) Single operation on single doc is by default Atomic.
2) read/write concerns defined on transaction level, not per operation
3) transactions wants to read from primary (readPreference:Primary)
4) Each transaction lasts 60 sec (could be changed with transactionLifetimeLimitSeconds flag)
5) Limit of 16Mb related to the doc, not for transaction


// ***************************** __ Internal execiution __  ***********************************
1) Create session and call startTransaction
2) Any operations comes to the server with session
3) commitTransaction & changes goes to the Oplog
4) In case any of error - call abortTransaction and all changes in memory deleted from memory


// ***************************** __ ACID vs BASE __  ***********************************
CAP theorem = you can select only 2 from 3
    Consistency
    Availability
    Partition Tolerance

ACID = C from CAP
    Atomicity - all or nothing. Any error = full rollback.
    Consistency - for each operation it pass through validation schema if exists
    Isolation - 2 operation can be executed on the same document. Mongo immediatly set WriteConflict
    Durability - commit executes with writeConcern: { w: "majority" }

BASE = A from CAP
    Basically Available - grant response for any request, but could be with error due to data inconsistency
    Soft state - data could change the state (on replicas) due to syncronization
    Eventual consistency - sometimes, all the system will be in consystent state

In Mongo if you works with:
    - transactions = ACID (due to w : majority)
    - replicas  = BASE 

// ***************************** __ USING transactions __  ***********************************
2 ways to use transactions:
    1) No retry logic (Core API)
        const session = client.startSession()
        try {
        session.startTransaction({
            readConcern:  { level: "snapshot" },
            writeConcern: { w: "majority" },
            maxCommitTimeMS: 2000
        })

        await accounts.updateOne(
            { _id: fromId },
            { $inc: { balance: -amount } },
            { session }                             // each operation comes with session
        )
        await accounts.updateOne(
            { _id: toId },
            { $inc: { balance: amount } },
            { session }                             // each operation comes with session
        )

        await session.commitTransaction()           // commit transaction
        } catch (err) {
            await session.abortTransaction()        // rollback transaction
            throw err
        } finally {
            await session.endSession()              // anyway closing session
        }

    2) with retry logic (Callback API) - retries will be done by driver automatically
        const session = client.startSession()
        try {
            await session.withTransaction(async (session) => {              // withTransaction grant automatic processing of errors & retries
                await accounts.updateOne({ _id: fromId }, { $inc: { balance: -amount } }, { session })
                await accounts.updateOne({ _id: toId },   { $inc: { balance: amount }  }, { session })
            }, {
                readConcern:  { level: "snapshot" },
                writeConcern: { w: "majority" }
            })
        } finally {
            await session.endSession()
        }

// ***************************** __ Error Labels __  ***********************************
Mongo set labels for errors to explain what to do with transaction that failed:
    - TransientTransactionError (Write Conflict/failover primary/Net issue) = retry starting from startTransaction
    - UnknownTransactionCommitResult = retry starting from commitTransaction
    - TransactionTooLargeForCache = no need to retry, need to decrease size of transaction

Example:
    async function runWithRetry(fn) {
        let attempts = 0
        while (true) {
            const session = client.startSession()
            try {
                session.startTransaction({ readConcern: { level: "snapshot" }, writeConcern: { w: "majority" } })
                await fn(session)

                // Retry only commit for UnknownTransactionCommitResult
                while (true) {
                    try {
                        await session.commitTransaction()
                        return
                    } catch (err) {
                        if (err.hasErrorLabel("UnknownTransactionCommitResult") && attempts < 5) {
                            attempts++; 
                            continue
                        }
                        throw err
                    }
                }
            } catch (err) {
                await session.abortTransaction().catch(() => {})
                // Retry from startTransaction for TransientTransactionError
                if (err.hasErrorLabel("TransientTransactionError") && attempts < 5) {
                    attempts++; 
                    continue
                }
                throw err
            } finally {
                await session.endSession()
            }
        }
    }

// ***************************** __ RW CONCERNS __  ***********************************
1) w != 0 (you need always to confirm write. Recommended w = "majority")
2) readConcern: "snapshot" = the best choise to grant reading true data even on sharded hosts. Have to come with w = "majority"!!!
3) If you create a new collection inside transaction you need readConcern: "local" (the most weaked type)
4) Inside transaction you can't read/write in 3 started DB (config, local, admin), can't write into system.* DBs, do any DDL actions(except creation collection).
    In case you'll do something that you can't you'll get OperationNotSupportedInTransaction error.


// ***************************** __ OpLog __  ***********************************
Special capped collectin in db = local
Internally when you do write operation:
    1) You write into your target collection
    2) You write into oplog the instruction of your change

Cause by default capped collection has limit, the time of records life in OpLog (capped collection) is limited.
It's Oplog Window. 

Commands:
    rs.printReplicationInfo()                                   = show oplog in Mb.
    db.adminCommand({ replSetResizeOplog: 1, size: 10240 })     = set oplog size to 10Gb

    use local
    db.oplog.rs.find({ op: "u" }).sort({ $natural: -1 }).limit(1).pretty() = check for last update operation (see example below)

Example:
    Command = db.collection.updateOne({ _id: 1 }, { $inc: { x: 5 } })
    oplog recooord:
        {
            "lsid": { ... },            // session id
            "txnNumber": Long("..."),   // transactiion number (if exists)
            "op": "u",                  // operation type: 'u' = update
            "ns": "db.collection",      // Namespace: database.collection name
            "ui": UUID("..."),          // UUID of collection
            "o": {                      
                "$v": 2,                // log version for difference (all inside the "o" object)
                "diff": {               // difference between old and new state
                    "u": {              // 'u' = 'update'
                        "x": 5          // set field x equal to 5
                    }
                }
            },
            "o2": { "_id": 1 },          // document that had been changed
            "ts": Timestamp(1712345, 1), // time of execution in seconds, segment inside this second
            "v": Long("2"),              // log version for full oplog record
            "wall": ISODate("...")       // real time of execution in ISO format, depend on server
        }
